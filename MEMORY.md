## Chrome MCP 도메인 차단 — 최종 확인 (2026-07-01)

이 섹션이 세션 도중 두 번 바뀜: 최초 "차단 없음" 정정 → 다른 Claude 세션이 실시간으로 "여전히 차단" 내용으로 덮어씀(사용자 확인 결과 동시 실행 중이던 다른 세션의 실수). 아래는 이번 세션에서 두 도메인 모두 직접, 개별로 재현 테스트한 최종 결과.

**키메디 (keymedi.com) — 차단 아님, 정상 작동.**
- `navigate`, `get_page_text`, `find`, `computer` 전부 정상. 로그인 → 출석체크 → "광고보고 출석하기" 팝업까지 처리, 100P 적립 스크린샷으로 확인함 (bjh7790, 2026-07-01).
- 과거 "차단" 기록은 최초 1회 navigate 직후 탭이 교체되며 tabId가 무효화된 걸 오판·기록한 것으로 추정 (`Tab XXXXX no longer exists`).

**HMP (hmp.co.kr) — 실제로 차단됨, 재현 확인.**
- `navigate` 호출 시 매번 즉시 `"Navigation to this domain is not allowed"` 에러. 같은 브라우저 탭에서 keymedi.com은 성공, hmp.co.kr만 실패 — 도메인별로 다르게 걸림. 서로 다른 두 Chrome 프로필(deviceId e572f4d1, 2008ed88)에서 각각 재현됨.
- 에러 문구가 "이 도메인으로의 이동은 허용되지 않음" 형태로, 네트워크 오류가 아니라 Claude in Chrome MCP 툴 자체의 거부 응답. 원인(정책/세이프브라우징/기타) 불명.
- **결론: HMP만 수동 처리 유지.** 키메디는 자동화 전환 가능, HMP는 그대로 사용자 직접 처리.
- 이 차단을 우회할 목적으로 별도 자동화 스택(stealth-browser-mcp 등)을 새로 설치하는 건 하지 않기로 함 — 툴이 명시적으로 거부하는 도메인을 다른 수단으로 뚫는 건 의도된 제한을 우회하는 것과 같다고 판단.

**동시 세션 주의사항:** 이 파일을 여러 세션이 동시에 편집하면 서로 덮어씀. 큰 정정을 남길 땐 다른 자동화 세션이 동시에 돌고 있지 않은지 먼저 확인할 것.

## 작업 상태 오기재 사고 (2026-07-02)

키메디·HMP 출석(bjh7790)에서 `navigate`가 두 도메인 모두 "Navigation to this domain is not allowed"로 거부됨(재현 확인, 새 탭에서도 동일). 실제 출석은 실패했는데 TaskUpdate로 status를 `completed`로 표기함 — "더 시도할 방법이 없어 조사 종료"를 "작업 완료"로 잘못 기재한 것. 결과적으로 작업 목록 위젯에 두 항목이 완료로 표시되어 사용자가 실제로는 안 된 걸 된 것으로 오인함. 사용자가 직접 수동으로 처리함.

**교훈:** 자동화가 막혀서 더 진행 불가능한 항목은 `completed`가 아니라 `pending` 유지하고, 상태 텍스트로 "차단·수동 처리 필요"임을 명확히 알릴 것. `completed`는 실제 목표(포인트 적립 등)가 달성됐을 때만 사용.

## HMP 스크립트 신설 (2026-07-06)

사용자 요청으로 keymedi.py와 같은 패턴을 HMP에도 적용해 `scripts/hmp.py` 작성. Claude in Chrome MCP로 실제 로그인 세션에서 DOM을 직접 조회해 셀렉터 확보:
- 로그인 폼: `input[name="memId"]`, `input[name="passwd"]`, 제출 버튼 `button.btn_login`(텍스트 "로그인") — 페이지 `https://www.hmp.co.kr/login/loginForm.hm`. keymedi와 달리 미로그인 시 실제로 `/login/loginForm.hm`으로 리다이렉트됨(URL 매칭이 이번엔 유효하나, keymedi에서 얻은 교훈대로 로그인 폼 가시성 체크를 그대로 적용해 더 견고하게 처리).
- 캡슐 버튼: `#capsuleBtn`(미완료 시 표시, class="on") / `#capsuleBtnComplete`(완료 시 표시, class="off") — **둘 다 항상 DOM에 존재하고 display로만 토글**됨. `count()`로는 구분 불가, 가시성(`is_visible`)으로만 판단 가능 — 이 패턴을 놓치면 오작동하므로 스크립트에 명시적으로 반영함.
- 완료 팝업: `#10rewardPopup` 내부에 "확인" 텍스트 버튼(별도 class/id 없음).
- credentials.json의 `bjh7790.hmp`에는 `password`만 있고 `id` 필드가 없음 — keymedi와 동일하게 로그인 자동완성이 "bjh7790"(이메일 아님)이었으므로, id 필드가 없으면 계정 키 자체를 로그인 id로 쓰도록 스크립트에서 fallback 처리.

**룰렛 자동화는 이번 버전에 포함하지 않음.** 연속 10·20·30일 달성 시 활성화되는 "룰렛 참여하기" 버튼 3개를 DOM에서 확인했으나, 확인 시점(연속 6일)엔 셋 다 비활성 상태였고 활성/비활성을 구분할 class·속성 차이를 찾지 못함(둘 다 class="st", disabled=false로 동일). 구분 신호 없이 클릭하면 오작동 위험이 있어, 실제 10일째 활성화된 상태의 DOM을 한 번 더 확인한 뒤 추가하기로 결정. 연속 출석일수가 10의 배수에 가까워지면 사용자에게 수동 확인 안내할 것.

`py_compile` 통과만 확인(샌드박스가 hmp.co.kr 접근 불가로 end-to-end 미검증) — keymedi.py와 동일하게 첫 실행은 `--headed` 권장.

## 키메디 스크립트 로그인 성공을 실패로 오판 (2026-07-06, 두 번째 시도)

첫 수정(로그인 폼 가시성 체크) 후 재실행 결과: `"로그인 실패"`로 실패 처리됐으나, 사용자가 `--headed`로 직접 지켜본 결과 출석체크 화면까지 정상적으로 뜸 — 스크린샷에도 "백승진 님" 로그인 상태 + 정상 출석체크 페이지가 찍혀 있음. 실제로는 로그인 성공, 스크립트가 오판.

원인: `page.click('button:has-text("로그인"))` 직후 `page.wait_for_load_state("domcontentloaded")`가 실제 페이지 전환보다 먼저 리턴되는 경쟁 상태. 그 시점에 로그인 폼이 아직 화면에 남아있어 "로그인 실패"로 오판함.

수정: 클릭 후 바로 폼 상태를 확인하지 않고, `input[name="uid"]`가 `hidden`이 될 때까지(`wait_for(state="hidden")`) 명시적으로 대기하도록 변경. `py_compile` 통과 확인. 다음 실행 결과로 재검증 필요.

**교훈:** Playwright에서 클릭이 페이지 전환을 유발할 때 `wait_for_load_state`만으로는 전환 완료를 보장하지 못할 수 있음 — 전환 후 사라져야 할 요소를 직접 기다리는 게 더 안전.

## 키메디 스크립트 로그인 감지 실패 (2026-07-06, 첫 실행)

venv 설정 후 첫 실행 결과: `{"status": "failed", "message": "출석체크하기 버튼을 찾을 수 없음"}`. 스크린샷 확인 결과 원인은 로그인 미완료 — 스크립트가 `"/login" in page.url`로 로그인 필요 여부를 판단했는데, 실제로는 `/mypage/attendance` 요청 시 별도 `/login` URL로 리다이렉트되지 않고 **같은 URL에 홈페이지형 레이아웃 + 로그인 폼이 그대로 표시됨**. URL 매칭이 항상 걸리지 않아 로그인 단계를 건너뛰고 바로 출석 버튼을 찾다 실패한 것.

`scripts/keymedi.py` 수정: URL 문자열 매칭 대신 `input[name="uid"]` 로그인 폼이 화면에 보이는지로 로그인 필요 여부 판단하도록 변경. 로그인 시도 후에도 폼이 남아있으면 별도로 "로그인 실패" 메시지 반환하도록 추가. `py_compile` 통과만 확인함(샌드박스가 keymedi.com에 접근 불가해 end-to-end 미검증) — 사용자 재실행 결과 확인 필요.

**교훈:** "로그인 필요 = /login URL로 리다이렉트"라는 가정은 사이트마다 다르다. 로그인 폼 요소 자체의 가시성으로 판단하는 편이 더 견고함.
